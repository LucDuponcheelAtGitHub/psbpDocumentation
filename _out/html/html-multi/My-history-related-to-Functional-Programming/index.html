2. My history related to Functional Programming

I am a mathematician.

I did mathematics research on Non-archimedean induced representations of compact zerodimensional groups.

That being said, I soon realized that, those days, in Belgium, mathematics could mainly be done as a backyard ritual, so, in order to earn some money for a living, I decided to become a programmer. I started doing computer science research as a late at night hobby. As such, I naturally became interested in the theory of functional programming.

I studied function level programming systems, as implemented in FP programming system.

I also studied a book about acceptable programming systems. I cannot easily find a reference to it.

Anyway, the latter study culminated in a paper Acceptable functional programming systems, written together with my twin brother Marc, who is also a mathematician. Among others, the paper illustrated acceptable functional programming systems using code written using the Pascal programming language). Those days, the Pascal programming language already, somehow, partially supported functional programming.

Both acceptable functional programming systems and function level programming systems are examples of pointfree effectfree functional programming systems.

The first functional programming language I used was Miranda). Miranda was, and still is, a perfect language to start learning effectfree functional programming with.

But programming is also about effectful programming ... .

The next functional programming language I used was Gofer). Gofer was the first functional programming language supporting type constructor classes. That capability of Gofer was, for me, the very reason to use it. I wanted to write code using with category theory related concepts, to deal with side effects.

This culminates in papers

  1. Composing monads,

  2. On the expressive power of Constructor Classes,

  3. Deterministic, Error-Correcting Combinator Parsers, and

  4. Using catamorphisms, subtypes and monad transformers for writing modular functional interpeters.

The first paper was, as a late night hobby, written together with Mark P. Jones, the author of Gofer. What is special about that paper is that we had never physically met each other. Those were the early days of the internet. We wrote the paper (and the code that came with it) together by sending emails to each other. In fact, it turned out to be an efficient way to work together. While one of us was sleeping (in Europe resp. the United States of America) the other one was working (in the United States of America resp. Europe). The paper also contained proofs. Those days they were simply lists of expressions representing equetional reasoning steps.

The other papers were written whilst working two years at the University of Utrecht. A unique experience for me for which I am forever greatful to Doaitse Swierstrsa (†), and Erik Meijer, who introduced me to Doaitse. Apart from being an exceptional computer scientist, Doaitse was the best manager I have ever worked with. Erik does not need any introduction. He is a living legend. Also Grahan Hutton worked at the University of Utrecht those days. What a team!

The second paper was written together with Erik Meijer and was presented at a conference in Glasgow, those days, and still today, an important centre of functional programming (cfr. Philip Wadler and Simon Peyton Jones).

The third paper was written together with Doaitse Swierstra and turned out to be a motivation for two monad specification related specifications: Programming with Arrows and Applicative Programming with Effects.

The fourth paper has never been published. It was based on Monad transformers and modular interpreters. I used catamorphisms and subtypes for making interpreter fragments more reusable. Anyway, the paper, somehow, turned out to be available online and has been cited several times.

The monad specification and applicative specification are examples of pointful effectful functional programming related specifications. The arrow specification is an example of a pointfree effectful functional programming related specification.

The program specification of this document is a pointfree effectful functional programming related specification that is similar to, but more powerful than the arrow specification. Think of programs as arrows with alternatives.

All this brings me to the motivation of this document.

The functor, applicative and monad specifications specify computation capabilities. Think of computations as effectful expressions. They are operational artifacts. They do not really have a meaning in the mathematical sense. Functions, on the other hand, are denotational artifacts. They do have a meaning in the mathematical sense. This document refers to effectful function specifications as program specifications, specifying program capabilities. Think of programs as effectful functions. They are denotational artifacts. They have a meaning in the mathematical sense.

In my opinion it is, as a programmer, more natural to think denotationally (what) than to think operationally (how).

Let's try to illustste this with some Lean code.

The computation related type class Bind has a member

bind :
  {α β : Type} →
  computation α → (α → computation β) → computation β

with law (>>= is infix for bind)

bind_assoc
  (cα : computation α)
  (αfcβ : α → computation β)
  (βfcγ : β → computation γ) :
    cα >>= αfcβ >>= βfcγ = cα >>= λ α => αfcβ α >>= βfcγ

The program related type class Sequential has a member

andThen
  {α β γ : Type} :
  program α β → program β γ → program α γ

with law (>=> is infix for andThen)

andThen_assoc
  (αpβ : program α β)
  (βpγ : program β γ)
  (γpδ : program γ δ) :
  ((αpβ >=> βpγ) >=> γpδ) = (αpβ >=> (βpγ >=> γpδ))

From a syntax point of view, I think that the Sequential member and law is more elegant than the Bind mamber and law. From a semantic point of view, how to explain bind and bind_assoc, and how to explain andThen and andThen_assoc?

Let's first deal with Sequential. I think of a program as transforming an argument to a result. andThen can be explained as: if you can transform argument α to result β, and you can transform argument β to result γ, then you can transform argument α to result γ. andThen_assoc can be explained as: first transforming argument α to intermediate result γ using αpβ >=> βpγ, and then transforming argument γ to final result δ using γpδ which is the same as the final result obtained by first transforming argument α to intermediate result β using αpβ, and then transforming argument β using βpγ >=> γpδ. Hopefully this is a meaningful denotational explanation.

Let's second deal with Bind. I think of a computation as, when executing it, yielding a result. bind can be explained as: binding result α, yielded by executing inner computation cα, to outer computation valued function αfcβ, yields outer computation cβ (that, when executing it, yields result β). bind_assoc can be explained as binding result α yielded by executing inner computation cα, to intermediate computation valued function αfcβ, yielding intermediate computation result cβ that, when executing it, yields intermediate result β that, when binding it to outer computation valued function βfcγ, yields outer computation result cγ which is the same as the outer computation result obtained by binding result α yielded by executing inner computation cα to the outer computation valued function binding α to the computation result obtained by binding result β, yielded by executing intermediate computation result αfcβ α to outer computation valued function βfcγ. Hopefully this is a meaningful operational explanation.

Let's face it, the denotational explanations are more elegant than the operational ones.
